Verdict

Salvageable prototype — not a rewrite-from-zero. Protocol, Pixi client loop, and registered server systems are live signal. Unwired assets and half-finished features are backlog, not trash. Delete only superseded cameras, build junk, unused deps, and pure duplicates.

Biggest wins before CI: wipe public/site/, delete superseded camera/cull code + unused deps, collapse or absorb ioengine (it imports game packets), unify the triple Typia toolchain — and leave art, map/decor configs, minimap, and ground-item scaffolding alone.

Simplify · high impact

Collapse over-engineering

1. Absorb or kill the fake package boundary: ioengine

~2k LOC of custom ECS + networking that pretends to be reusable, but game_object.ts imports @shared/packet_definitions. Engine depends on game — wrong direction. For a rework, fold it into server/engine + shared/utils rather than polishing a package that isn’t one. 

Today (leaky)
src/ioengine/          ← “engine”
  └── imports @shared  ← game protocol

Target
packages/shared/       protocol, ids, math
packages/server/       ECS + systems + net
packages/client/       Pixi + UI
(no engine→game imports)

2. Collapse Typia to one path (or drop it)

Three overlapping mechanisms for one job:

  • preload.ts + bunfig.toml → bun-plugin-typia
  • build.ts → @ryoppippi/unplugin-typia
  • ts-patch + typia/lib/transform in tsconfig + prepare script

Pick one for CI (likely Bun plugin only, or replace with Zod/valibot and delete the Typia stack). Typia is the heaviest tooling tax in the repo.

3. Thin wrappers & duplicates

  • src/ioengine/client/index.ts — 2-line re-export barrel
  • src/game/client/network/serializer.ts — 6-line wrapper
  • Duplicate id_map loaders: client configs/id_map.ts ↔ server loaders/id_map.ts → one shared module
  • Duplicate BasicPoint type definitions
  • Mixed import styles: aliases vs deep relatives vs .js suffixes — pick one convention

4. Scripts & docs that lie

  • package.json "module": "index.ts" — file does not exist
  • README.md says bun run index.ts — wrong; real flow is two processes
  • No dev / typecheck / lint / test scripts
  • jsx: "react-jsx" in tsconfig with zero React / no .tsx

Target architecture for the rework

Intent of the current folders is mostly right; enforcement and packaging are not. 

Proposed layout
packages/
  shared/     packets, ids, types, pure math
  server/     ECS, systems, configs, WS
  client/     Pixi, UI, assets, build entry
apps/ or root scripts:
  static host (optional; CDN in prod)
  game server

Rules
• shared has zero imports from client/server
• server may import shared only
• client may import shared only
• enforce with package exports + eslint import boundaries
• one tsconfig project references graph
• one validation library, one bundler path
Map pipeline: keep map.json + load_map.ts. Point the server entry back at the loader when ready; until then procedural loadTestMap() is fine as the active bootstrap. Do not treat the unused map as dead weight.

CI readiness — what’s missing

There is no .github/, no test runner, no linter, no typecheck script. Add CI only after the orphan purge so the first green build isn’t fighting dead code.

Recommended phases

  1. Hygiene PR — strip only true junk

    Wipe public/site, delete superseded cameras/cull, unused deps, duplicate types, fix README + package.json module field. Leave assets, map/decor configs, and WIP UI alone.

  2. Tooling PR — CI skeleton

    One tsconfig, one Typia path (or remove), add typecheck + build GitHub Action, enable noUnusedLocals carefully around parked WIP, add Biome/ESLint.

  3. Boundary PR — architecture

    Invert/remove engine→game imports; move shared utils; introduce package (or folder) boundaries with lint enforcement. Split bootstrap god files lightly.

  4. Content wiring — near-future features

    Reconnect map loader, wire decor + unmapped sprites, revive minimap/ground items, finish crafting categories. Then add tests around packets/collision/config load.
